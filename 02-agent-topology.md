# 02 - Agent Topology（Agent 角色拓扑）

## 结论
默认采用 **2+1 拓扑**：
- **Planner**：初始化 / 任务建模 / 拆分 / 验收定义
- **Builder**：实现 / 测试 / 文档 / 执行记录
- **Reviewer**（按需或常驻）：独立审查 / 风险识别 / 返工建议

不建议一开始采用 4+ 执行角色。原因不是“做不到”，而是**收益通常小于编排复杂度**。

---

## 1. 角色设计原则

### 1.1 依据“目标函数差异”拆角色，而不是按组织架构拟人化
只有当两个职责在以下维度有明显差异时，才值得拆分：
- 输入上下文不同
- 成功标准不同
- 决策偏好不同
- 需要独立立场互相制衡

### 1.2 编排器不等于普通 Agent
建议把“编排/调度”视为 **Harness + 人类主控** 的职责，而不是再造一个高频执行的 Manager agent。

### 1.3 Prompt 要薄，规则要厚
- Prompt：只承载角色差异与本轮目标
- 项目规则：落到文件（GOVERNANCE / TASK / RUN LOG / ADR）
- 交接：用结构化模板，而不是依赖会话记忆

---

## 2. 三种常见拓扑对比

## A. 单 Agent
### 适用
- 短任务
- 一次性脚本
- 低风险改动

### 问题
- prompt 过于臃肿
- 自审偏差大
- 规划、实现、审查互相污染
- 长周期容易漂移

### 结论
仅适合简单任务，不适合持续推进型工程项目。

---

## B. 双角色（Initializer + Builder）
### 适用
- harness 初期
- 项目尚小
- 人类愿意承担部分 review

### 优点
- 简洁，最容易落地
- 明显优于单 Agent

### 缺点
- 质量审查可能不足
- Builder 容易“自己证明自己是对的”

### 结论
是很好的起步方案，但通常应向 2+1 演进。

---

## C. 2+1（Planner + Builder + Reviewer）
### 优点
- 拆分足够有效，但复杂度仍可控
- 规划、执行、审查三种思维模式分离
- 最适合长周期、高质量开发

### 结论
**推荐默认方案**。

---

## D. 4+ 多角色
例如：Planner / Architect / Developer / Tester / Release / Security

### 适用前提
- 任务量大且并行度高
- 交接协议成熟
- 质量门与自动化较完整
- 项目规模足以抵消编排成本

### 风险
- Prompt 维护困难
- 状态同步成本高
- 自动流转失败概率高

### 结论
后期再考虑，不适合作为起点。

---

## 3. 推荐的职责边界

## 3.1 Planner
### 负责
- 把模糊目标转为任务卡
- 定义范围（In / Out）
- 明确依赖、风险、回滚
- 制定验收标准
- 判断是否可以进入执行态

### 不负责
- 大量编码实现
- 跳过验收定义直接推进

### 核心判断标准
> 一个任务若无法被清晰验证，就不应进入执行。

---

## 3.2 Builder
### 负责
- 在任务卡约束下实现最小增量
- 执行测试与验证
- 更新运行日志
- 输出可 review 的变更摘要

### 不负责
- 擅自扩范围
- 重新解释需求替代规划
- 无验证宣布完成

### 核心判断标准
> 完成不是“代码写完”，而是“证据齐全且可交接”。

---

## 3.3 Reviewer
### 负责
- 独立评估任务是否真正达成
- 识别回归风险、耦合、遗漏测试、文档不一致
- 判断是否退回 Rework
- 提出规则与测试改进建议

### 不负责
- 在未明确授权下大规模重写实现
- 代替 Planner 重定义目标

### 核心判断标准
> 默认寻找缺口，而不是默认相信交付方。

---

## 4. 推荐会话布局（适用于 OpenClaw / Claude Code / Codex 类环境）

### 最小配置
- 主会话：人类 + Harness 编排
- Planner：短会话或主会话承担
- Builder：持久线程 / 持久 session
- Reviewer：按需调用，复杂任务可持久化

### 原因
- Builder 最依赖长期上下文
- Reviewer 需要独立视角，但触发频率低于 Builder
- Planner 更像阶段性初始化与澄清工作

---

## 5. 升级信号

### 从双角色升级到 2+1
当出现以下任一信号：
- Builder 自证通过但返工频繁
- 任务跨模块、跨包、跨接口边界增多
- 测试证据不足导致上线/合并风险升高
- “完成”的定义在团队中频繁不一致

### 从 2+1 升级到 4+
当出现以下组合信号：
- 并行任务数显著增加
- 单 Planner 成为瓶颈
- 发布/运维/安全有独立上下文与门禁
- 已有成熟任务交接标准与自动化状态同步

---

## 6. 最终建议
对当前 `robot-controller` 及未来类似项目，推荐：

### 默认执行架构
- **Planner**：任务建模、拆分、验收、风险
- **Builder**：实现、测试、文档、日志
- **Reviewer**：独立审查、返工建议、规则补强

### 展示层命名（可选）
如果保留你原来的团队人格设定：
- 贾诩 ≈ Planner / Architecture
- 赵子龙 ≈ Builder
- 小乔 ≈ Reviewer / Tester
- 阿瞒 ≈ 人类主控 / 编排视角

这样可以保留协作风格，但执行架构仍保持工程上的克制与稳定。
